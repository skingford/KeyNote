category: Go基础
title: G-M-P 并发模型解析
---
请简述 Go 语言中的 G-M-P 调度模型，以及 P 的作用是什么？
====
- **G (Goroutine)**：代表一个用户态的轻量级线程，包含了自己的栈和指令指针。
- **M (Machine)**：代表操作系统的物理/内核级线程，由操作系统调度和管理。
- **P (Processor)**：代表处理器/逻辑运行上下文，是调度器的核心。

**P 的核心作用**：
P 充当了 M 和 G 之间的桥梁。它包含了自己的本地可运行 G 队列 (Local Queue)。
只有当 M 获取到了 P 之后，M 才能执行 G 上的代码。这种设计主要是为了减少全局锁竞争（如果直接使用全局队列，多个 M 并发获取 G 就会频繁竞争锁），同时 P 还做了一些工作窃取 (Work Stealing) 来平衡不同 M 之间的负载。

category: Go基础
title: defer 的执行顺序和性能开销？
---
Go 中的 `defer` 语句是按照什么顺序执行的？在 Go1.14 之后 `defer` 性能有哪些改进？
====
`defer` 的执行顺序是**后进先出 (LIFO/栈结构)**。即最后声明的 defer 会最先被执行。通常在函数返回前触发。

**性能改进**：
- **Go 1.13 之前**：在堆上分配，并在挂载时产生较大的开销。
- **Go 1.14 规范了 Open-Coded Defer**：对于没有循环的 `defer`，编译器会在编译阶段直接把 `defer` 的执行代码直接插入到每个函数 return 指令之前。从堆分配变为了栈分配或直接内联，将 `defer` 的运行开销降低到了几乎与直接调用函数无异。

category: 微服务架构
title: 分布式理论之 CAP 定理
---
请解释 CAP 定理的三个字母分别代表什么？为什么现实系统中通常会牺牲 C 来换取 A？
====
- **C (Consistency - 一致性)**: 所有节点在同一时间具有相同的数据。
- **A (Availability - 可用性)**: 保证每个请求都能收到非错的响应，但不保证获取的是最新数据。
- **P (Partition tolerance - 分区容错性)**: 系统中任意信息的丢失或失败不会影响系统的继续运作。

在分布式架构中，网络分区 (P) 是不可能完全避免的（比如机房断网、光缆故障）。
一旦发生了网络分区，我们必须在 **C（一致性）** 和 **A（可用性）** 之间做出选择。
在互联网高并发 C 端业务中，通常系统会选择 **AP**（保证服务可用，接受最终一致性），因为停机或拒绝访问带给用户的体验损失远大于短时间内看到陈旧数据。对于金融系统，往往选择 **CP**，宁可服务报错也不允许资金计算不一致。

category: 数据库优化
title: MySQL: B+ Tree vs LSM Tree 的优缺点
---
请对比目前主流存储引擎中的 B+ 树与 LSM（Log-Structured Merge-Tree）树在读写性能和架构上的区别。
====
| 特性对比 | B/B+ Tree (如 InnoDB) | LSM Tree (如 RocksDB, HBase) |
| :--- | :--- | :--- |
| **基础结构** | 层级树状结构，叶子节点构成单向/双向链表。 | 内存表(MemTable) + 磁盘层级(SSTable)，顺序写入合并。 |
| **写入性能** | 较差，需要寻找特定页更新，容易引起**页分裂**(Page Split)与合并。通常为**随机写**。 | **极高**。追加写(Append Only)到日志和内存表，批量刷盘，变为**顺序写**。 |
| **读取性能** | **极佳**。通常只要 O(log N) 次甚至一两次磁盘 I/O 就能命中目标。范围查询友好。 | 较差。查找时可能需要跨多个 SSTable 层进行搜索，通常需要布隆过滤器(Bloom Filter)来优化读放大。 |
| **空间放大/碎片** | 存在页碎片问题，空间放大。数据更新也是原地修改(In-place update)。 | 采用后台 Compaction 不断清理旧版本数据，但在特定时期也存在空间翻倍风险。 |
| **典型适用场景** | 读取密集型，传统关系型数据库（读写比很高的业务）。 | 写入密集型，日志流，时间序列数据库或海量 KV 存储。 |

category: 分布式协议
title: Raft 算法中的角色与选主逻辑
---
在 Raft 分布式一致性协议中，节点有哪几种主要角色？简述其选主(Leader Election)过程。
====
**三种阶段角色**：
1. **Leader (领导者)**：负责处理客户端的所有请求，并将日志状态同步给所有的 Followers。同一时期只有一个。
2. **Follower (跟随者)**：被动节点，不会主动发送任何请求，只响应 Leader 和 Candidate。
3. **Candidate (候选者)**：Follower 到 Leader 过渡的角色。

**选主过程简述**：
1. **心跳超时**：每个 Follower 内部有个随机的选举超时时钟（比如 150-300ms）。若超时未收到 Leader 心跳，便转变自己为 Candidate，自增当前 Term（任期），并为自己投票。
2. **请求投票**：Candidate 随后向集群内的其他节点发送 \`RequestVote\` RPC 请求。
3. **节点响应**：其他 Follower 如果之前在这个 Term 里还没投过票，并且 Candidate 的日志至少和自己一样新，就给它投一票并重置自己的超时定时器。
4. **成为 Leader**：当一个 Candidate 获得了集群中**半数以上 (`N/2 + 1`)** 的选票后，它就会升级为 Leader，并立刻向其他节点广播心跳包，压制其他节点发起选举。

category: Go基础
title: Go 中的 Map 是并发安全的吗？
---
如果多个 Goroutine 同时读写一个原生的 \`map[string]int\` 会发生什么？如何解决这个问题？
====
**原生的 Go map 不是并发安全的**。

如果发生了多个 Goroutine 同时对同一个原生 map 执行读写或同时执行写操作，Go 运行时会直接抛出 **`fatal error: concurrent map read and map write`** 或被检测到 `concurrent map writes` 而发生致命的 panic，导致程序崩溃（且这种 panic 无法被 `recover` 捕获）。

**解决办法**：
1. **轻量级/自建锁**：使用 `sync.Mutex`（互斥锁）或 `sync.RWMutex`（读写锁）在访问 map 的外部进行包裹封装。适用于写操作不极其密集、锁粒度可以接受的场景。
2. **高级/官方方案 `sync.Map`**：标准库提供的并发安全 Map。
   - 内部采用了读写分离的思想，维护了 `read` (只读不加锁) 和 `dirty` (读写需加锁) 两个原生字典。
   - 它做了大量的冗余和指针技巧来优化性能，**非常适合“读多写少”** 或者“不同 goroutine 各自修改不同的 keys”的场景。若频繁插入新 key，性能不如简单加一把 RWMutex。
